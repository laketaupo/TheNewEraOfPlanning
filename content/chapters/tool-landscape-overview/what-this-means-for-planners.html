System literacy is not an IT skill that happens to be useful for planners. It is a core planning competency — as fundamental as knowing how to read a supply coverage chart or run an exception report. Planners who lack it make predictable, repeatable mistakes: they trust a number without knowing when it was last refreshed, they try to fix a problem in the wrong system, or they escalate to the wrong person and lose half a day. Being system-literate does not mean knowing how to configure or maintain the systems. It means knowing where each number comes from, how old it might be, and who owns it. That knowledge is the difference between diagnosing a discrepancy in five minutes and spending a day investigating the wrong thing.

The First Question: Which System Is This From?

Every number you look at in Planning Software originated somewhere else. It did not appear there on its own — it was calculated or carried over from one of the three source systems: ERP, MDM, or FMS. Developing the habit of tracing a number back to its source before reacting to it is one of the highest-leverage skills a planner can build.

Consider what happens when a demand figure looks too high. Before you do anything else, you need to know what that number actually is. If it is a statistical forecast generated by Planning Software, the right response is to examine the statistical model — its history, its smoothing parameters, any recent demand events that might have inflated it. If it is a commercial overlay entered by a sales manager, the right response is a conversation with that person about what they know that the model does not. But if it is a firm customer order that has already been confirmed and imported from ERP, the right response is to do nothing with the demand number at all — it is not a forecast, it is a commitment. The same number, in three different situations, requires three completely different responses. System literacy tells you which situation you are in.

The same logic applies to supply. A planned order quantity that looks wrong could have several causes. Has the lot size for that item been set up correctly in MDM? Lot sizes are master data — they live in MDM, they flow into Planning Software, and if Planning Software is generating orders that seem systematically over- or under-sized, the first check is whether the MDM record reflects the current commercial or operational reality. Has there been a BOM change recently? A reformulation or packaging change that was actioned in MDM but not yet fully propagated would cause Planning Software to plan against the old structure, which can produce nonsensical results. These are not Planning Software problems. They are MDM problems that are being surfaced through Planning Software.

Inventory discrepancies follow the same pattern. If the on-hand quantity in Planning Software does not match what the warehouse is reporting, the investigation starts in ERP. Has the goods receipt been posted? Has a write-off or adjustment been processed? Planning Software is displaying a copy of what ERP holds — if the copy looks wrong, the question is what ERP actually says, and whether ERP has had time to reflect what physically happened. Changing the inventory figure in Planning Software directly would be fixing the copy rather than the original, and the next integration run would overwrite the change anyway.

Understanding Data Latency

Planning Software is never completely current. It reflects the state of ERP, MDM, and FMS as of the last time data was pushed from those systems — and that happens on a schedule, not in real time. Most planning environments run an overnight batch that refreshes the data landscape: inventory positions, open orders, master data changes, field supply updates. If that batch runs at 2am and you start work at 7am, you are working with a five-hour-old picture of the world. In most cases, that is perfectly acceptable — five hours overnight is not much time for the world to have changed.

But the assumption "the batch ran and it worked" is one that planners learn to verify rather than take for granted. A batch job can fail silently. There may be no alert, no warning message on your screen, no obvious indication that something went wrong. You open Planning Software, the data looks plausible, and you start making decisions. What you do not know is that you are working with yesterday's picture — or, if the previous batch also failed, the day before that. A 29-hour-old inventory position looks the same on screen as a 5-hour-old one.

Most planning environments provide a way to check integration health: a batch run log, a data freshness indicator, or a status dashboard. The exact location depends on the platform and how your environment has been configured. If you do not know where to find it, that is worth resolving now, not the next time something looks wrong. Checking it should be a morning habit — a 30-second verification that the overnight run completed successfully before you begin any planning work that depends on current data. Teams that build this check into their daily routine catch integration failures hours earlier than teams that do not.

When data latency is relevant to a decision — when you are trying to understand whether a supply shortage is real or an artefact of a delayed goods receipt, for example — the question to ask is not just "what does Planning Software show?" but "when was Planning Software last updated, and is that recent enough for this decision?" Sometimes the answer is that you need to go directly to ERP to get the current position, rather than waiting for the next batch.

Fix Things in the Right System

One of the most common — and most consequential — planner mistakes is correcting data in the wrong system. It is understandable: if you can see a number in Planning Software and you can see it is wrong, and there is a field that lets you change it, the path of least resistance is to change it there. But this creates a local override that will not survive the next integration run. When MDM pushes an updated version of that master record, or when ERP sends the next inventory snapshot, your change is overwritten. You have spent time on a fix that did not last.

The mental model that prevents this mistake is straightforward: source of record systems own the data. Planning Software displays a copy. Fixes go in the source of record.

If a lead time is wrong, the fix belongs in MDM. Not in Planning Software, not as a manual override on a specific order, but in the MDM record for that product. That way the correction propagates everywhere the lead time is used, and it persists across every future planning cycle. If a BOM is wrong, same principle — the master record in MDM needs to be corrected, and once it is, the correction flows downstream automatically.

If inventory in Planning Software does not match ERP, the question is not how to adjust Planning Software — it is whether the transaction in ERP has been posted correctly. A goods receipt that has been physically completed but not yet posted in ERP is a real inventory event that simply has not been recorded yet. The planning implication is real — you may need to temporarily factor in stock you know exists but that the system cannot yet see — but the resolution is the ERP posting, not a Planning Software adjustment.

Field supply figures from FMS carry their own timing logic. FMS estimates are updated on a cadence — weekly or monthly in most agricultural and contracted-supply environments — and the figure you see in Planning Software reflects the most recent FMS update, not a real-time field count. If a supply estimate has not moved in three weeks, that is almost certainly not a data problem; it is the expected update frequency. If you believe the field reality has shifted materially since the last update, the response is to contact the FMS owner or field team to request a re-forecast — not to adjust the number in Planning Software yourself.

Escalate to the Right Person

When something is genuinely wrong — not just unexpected, but actually wrong — the effectiveness of your escalation depends on knowing which layer of the system is the source of the problem.

A mismatch between Planning Software and ERP that persists after the overnight batch has completed is typically an integration problem. The data pipeline between the two systems is not working as expected — either it failed to run, ran but produced errors, or mapped data incorrectly. This is a question for the integration or IT team, who can inspect the job logs and identify what went wrong.

A planning parameter that is consistently wrong — a lead time that keeps reverting to an incorrect value, a safety stock rule that does not match current policy — is a data governance issue. The MDM record needs to be corrected, and the process for reviewing and maintaining that record needs to be assessed. This goes to the data manager or MDM team, not to Planning Software configuration.

A Planning Software recommendation that looks structurally wrong even when the underlying data is accurate — lot sizes that do not make sense, exception messages that fire on the wrong conditions, supply netting that does not match the expected logic — is a configuration or logic issue. This is a question for the Planning Software team or, in environments where the system was recently implemented, the implementation partner. It is rarely something a planner can resolve directly.

Knowing which of these three destinations applies to a given problem is not advanced knowledge — it is basic system literacy, and it is something every planner in a multi-system environment should be clear on from day one. A misdirected escalation does not just waste your time; it delays resolution for everyone downstream who is waiting on the same data.
