# Search Overlay — Pillar Pills Filter

**Date:** 2026-06-24
**Status:** Approved

## Problem

The search overlay's pillar filter renders as a dropdown list (auto-generated by PagefindUI). This is visually dated and hard to discover. The goal is to replace it with coloured pill buttons that are always visible below the search bar.

## Approach

**Option B — Custom pill UI using the raw Pagefind JS API.**

Keep PagefindUI for the search input and its default (unfiltered) result rendering. When a pill is active, hide PagefindUI's result list via CSS and render pillar-filtered results in a parallel `#filtered-results` container, driven by the raw `pagefind.js` module.

This avoids coupling to PagefindUI's internal DOM structure while reusing its input handling, debouncing, and keyboard navigation.

## Changes

All changes are confined to `src/components/Search.astro`.

## JavaScript Logic

### State

```js
let activePillar = null;   // null | 'technology' | 'process' | 'data' | 'people'
let pagefindRaw = null;    // raw pagefind module, loaded once after PagefindUI init
```

### Initialisation (inside existing `script.onload` handler)

After PagefindUI initialises, import the raw pagefind module (resolves from the browser cache PagefindUI already populated):

```js
pagefindRaw = await import('/pagefind/pagefind.js');
```

Wire pill click handlers and attach an `input` listener to the Pagefind-created `<input>` that re-runs filtered search when `activePillar` is set.

### Pill Click Handler

```
pill clicked
  if pill.dataset.pillar === activePillar:
    → clear activePillar
    → remove 'active' class from pill
    → remove 'filtering' class from #search-container
    → hide + clear #filtered-results
  else:
    → set activePillar = pill.dataset.pillar
    → swap 'active' class to clicked pill
    → add 'filtering' class to #search-container
    → run filtered search with current input value
```

### Filtered Search

```js
async function runFilteredSearch(query) {
  const panel = document.getElementById('filtered-results');
  if (!query || query.length < 2) {
    panel.innerHTML = emptyStateHtml(activePillar);
    panel.classList.remove('hidden');
    return;
  }
  const search = await pagefindRaw.search(query, {
    filters: { pillar: activePillar }
  });
  const results = await Promise.all(
    search.results.slice(0, 10).map(r => r.data())
  );
  panel.innerHTML = results.length
    ? results.map(renderFilteredResult).join('')
    : noResultsHtml(activePillar);
  panel.classList.remove('hidden');
}
```

### Result Rendering

Each result renders as an anchor row matching the existing PagefindUI `.pagefind-ui__result` visual style: same padding (`10px 12px`), border-radius (`10px`), bold title, excerpt text, indigo hover background.

```js
function renderFilteredResult(r) {
  return `<a class="pagefind-ui__result" href="${escape(r.url)}">
    <div class="pagefind-ui__result-inner">
      <p class="pagefind-ui__result-link">${escape(r.meta.title)}</p>
      <p class="pagefind-ui__result-excerpt">${r.excerpt}</p>
    </div>
  </a>`;
}
```

### Input Listener (debounced, 150 ms)

Attached to the Pagefind-created `<input>` after PagefindUI renders it (same MutationObserver already used for notes search). When `activePillar` is set, call `runFilteredSearch(input.value.trim())` on each input event.

### Edge Cases

| Scenario | Behaviour |
|---|---|
| Pill active, no query typed | `#filtered-results` shows "Type to search within [Pillar]" |
| Pill active, query has no results | `#filtered-results` shows "No results in [Pillar]" |
| Pill active, query < 2 chars | Same as no query — empty state |
| Modal closed and reopened | `activePillar` resets to `null`; pills return to inactive |
| Pagefind not loaded (dev mode) | Pills render but are non-functional; no JS error because `pagefindRaw` stays `null` |

### Notes Results

Unchanged. The notes search runs off the same input independently of `activePillar`.
